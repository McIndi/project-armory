# Keycloak State — Armory

> Keycloak is embedded in the BeeAI Agent Stack Helm chart (`agentstack-keycloak`). It provides identity and access management for the BeeAI platform.

---

## Deployment

| Setting | Value |
|---|---|
| Helm release | `agentstack` (bundled sub-chart) |
| Namespace | `agentstack` |
| Service | `agentstack-keycloak` · ClusterIP · port `8080` |
| Public URL (via Ingress) | `https://armory.local/realms/agentstack` |
| Ingress resource | `agentstack-keycloak` in `agentstack` ns |
| Ingress path | `/realms` (prefix) |

## Realms

### `agentstack` realm

This realm is created and managed by the BeeAI chart at deploy time. It is the only application realm in the current deployment.

| Setting | Value |
|---|---|
| Realm name | `agentstack` |
| Public issuer URL | `https://armory.local/realms/agentstack` |
| Admin console | `https://armory.local/realms/agentstack/...` (Keycloak admin path) |

### `master` realm

The built-in Keycloak master realm. The Keycloak admin console password is auto-generated by the chart and stored in the k8s Secret `keycloak-secret` (key: `admin-password`).

```bash
# Retrieve Keycloak admin console password
vagrant ssh -c "sudo k3s kubectl get secret -n agentstack keycloak-secret \
  -o jsonpath='{.data.admin-password}' | base64 -d; echo"
```

## Users (agentstack realm)

| Username | Role | How Created | Credential Source |
|---|---|---|---|
| `admin` | `agentstack-admin` | Chart seed via `auth.seedAgentstackUsers` Helm value | OpenBao KV `secret/beeai/credentials` → VSO → k8s Secret → Ansible set_fact → Helm value at deploy time |

The `admin` password flows: **OpenBao KV → VSO sync → k8s Secret `beeai-credentials` → Ansible reads `.data.admin_password` → passed to `keycloak.auth.seedAgentstackUsers[0].password` Helm value → chart creates/updates the user**.

User seeding is idempotent: re-running the playbook re-passes the same password from OpenBao, so the user is updated to match if it was changed manually.

## Clients (known, chart-managed)

The agentstack chart registers OAuth2/OIDC clients for its own services. The exact client IDs are determined by the chart. Expected clients:

| Client ID | Purpose |
|---|---|
| `agentstack` (or similar) | BeeAI API backend service |
| `agentstack-ui` (or similar) | BeeAI Next.js UI (NextAuth provider) |

The NextAuth callback URL is configured via `auth.nextauthUrl: https://armory.local` — Keycloak must allow this as a valid redirect URI. The chart sets this automatically via the seed configuration.

## Current Integration Points

| Integrator | How | What |
|---|---|---|
| **BeeAI UI** | NextAuth.js → OIDC | User login via `https://armory.local/realms/agentstack` issuer; `auth.nextauthUrl` and `auth.apiUrl` set to `https://armory.local` |
| **BeeAI API** | Bearer token validation | Validates JWTs issued by the `agentstack` realm |
| **Ansible** | Helm value `auth.seedAgentstackUsers` | Seeds admin user on every deploy; password sourced from OpenBao |
| **OpenBao** (indirect) | Via VSO + Ansible | Admin password stored in and retrieved from `secret/beeai/credentials` |

## Potential Future Integration Points

### Additional Realm Users / Groups
New users or groups can be seeded by extending the `auth.seedAgentstackUsers` array in `beeai_tofu_chart_values`. Passwords should be stored as additional keys in `secret/beeai/credentials` in OpenBao and synced via the existing `beeai-credentials-sync` VaultStaticSecret.

### Service-to-Service OAuth2 (Client Credentials)
BeeAI services or external tools can authenticate using Keycloak client credentials flow instead of user passwords. This requires creating a confidential client in the `agentstack` realm and storing the client secret in OpenBao at e.g. `secret/beeai/service-clients/<name>`.

### SSO with External Identity Providers (IdP Federation)
Keycloak supports federating an external IdP (Azure AD, Google, GitHub, LDAP, etc.) as an identity source for the `agentstack` realm. Users from the external IdP can be mapped to `agentstack-admin` or other roles via Identity Provider mappers. No infrastructure changes required beyond Keycloak realm configuration.

### Keycloak Admin API Access from Ansible
The Keycloak Admin REST API (`/realms/master/protocol/openid-connect/token`) can be called from Ansible to automate realm configuration (clients, roles, groups) beyond what the chart supports via Helm values. The admin password would be retrieved from OpenBao at runtime.

### Fine-Grained Authorization (UMA / Resource Server)
Keycloak's Authorization Services (UMA 2.0) can protect individual API resources with policy-based access control. This would allow different user roles to access different BeeAI agents or workspaces. Currently not in scope but the realm is already the correct boundary for this.

### Webhook / Event Listener
Keycloak supports Event SPI listeners. A custom listener could publish login/logout events to an external sink (log aggregator, SIEM) for audit purposes.

### Dedicated Admin Realm User
Currently the only seeded user is the application admin. A separate `keycloak-ops` user scoped to the `master` realm for infrastructure management would follow least-privilege principles for day-2 operations.

## Known Constraints

| Constraint | Detail |
|---|---|
| Chart-managed realm | The `agentstack` realm is owned by the Helm chart. Manual changes to clients or realm settings may be overwritten on the next `helm upgrade`. |
| Password update job disabled | `postgresql.auth.passwordUpdateJob.enabled: false` — this is a Bitnami workaround; the hook secret would persist across upgrades and cause failures. |
| Single-node Keycloak | No HA configuration; built-in H2 or bundled PostgreSQL. Keycloak shares the same PostgreSQL instance as the rest of agentstack. |
| Issuer URL must be reachable by clients | `publicIssuerUrl` is set to `https://armory.local/realms/agentstack`. Clients resolving this URL must have `armory.local` in their `/etc/hosts` or DNS, and must trust the Armory Root CA. |
